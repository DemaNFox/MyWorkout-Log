# AGENTS.md — правила для coding agent

## 1. Роль агента

Ты — senior React Native engineer и mobile architect.

Твоя задача — разрабатывать Android-приложение `Workout Logger` на React Native + TypeScript с production-like архитектурой.

Работай как инженер, который поддерживает проект долгосрочно, а не как генератор демо-кода.

## 2. Главная цель проекта

Создать минималистичный offline-first журнал тренировок.

Приложение решает только три задачи:

1. Спланировать тренировку.
2. Провести тренировку.
3. Показать динамику.

Если новая функция не помогает одной из этих задач — не добавляй её.

## 3. Запрет на расширение scope

Не добавляй без прямого указания владельца проекта:

- AI-тренера;
- рекомендации тренировок;
- соцсеть;
- друзей;
- ленту активности;
- достижения;
- магазин программ;
- питание;
- калории;
- воду;
- сон;
- шаги;
- вес тела пользователя;
- процент жира;
- фотографии прогресса;
- видео упражнений;
- огромную базу упражнений;
- push-уведомления;
- интеграции с часами;
- аккаунты;
- backend;
- облачную синхронизацию;
- черновик незавершённой тренировки.

## 4. Архитектура

Используй Feature-Sliced Design, адаптированный под React Native:

```text
src/
  app/
  pages/
  widgets/
  features/
  entities/
  shared/
```

Разрешённое направление зависимостей:

```text
app -> pages -> widgets -> features -> entities -> shared
```

Никогда не нарушай это направление.

## 5. Правила слоёв

### app

Только инициализация приложения:

- providers;
- navigation;
- db bootstrap;
- global styles/theme;
- error boundaries.

### pages

Только композиция экранов.

Не размещай в pages сложную бизнес-логику.

### widgets

Крупные UI-блоки из features/entities/shared.

### features

Пользовательские действия:

- create plan;
- start workout;
- complete set;
- import data;
- backup restore.

### entities

Бизнес-сущности, типы, репозитории и доменная логика:

- Plan;
- TrainingDay;
- PlannedExercise;
- WorkoutSession;
- WorkoutExercise;
- WorkoutSet;
- Analytics;
- Settings.

### shared

Переиспользуемая техническая база.

Shared не должен знать о бизнес-сущностях верхних слоёв.

## 6. TypeScript

Обязательно:

- strict mode;
- не использовать `any`, кроме технически неизбежных случаев;
- все navigation params типизированы;
- все repository methods типизированы;
- все import/export schemas типизированы;
- доменные статусы оформлять union types.

## 7. Хранение данных

Источник истины — локальная база данных.

Предпочтительно SQLite.

Правила:

- UI не работает с SQL напрямую;
- все запросы идут через repositories;
- схема меняется через migrations;
- история тренировок хранит snapshot названий;
- импортируемые данные проходят валидацию.

## 8. UX-правила

Приложение используется во время тренировки.

Приоритет:

- минимум кликов;
- крупные кнопки;
- быстрый ввод веса и повторений;
- прошлое значение видно сразу;
- таймер доступен быстро;
- офлайн-работа обязательна;
- без лишнего визуального шума.

## 9. Аналитика

Аналитика строится по упражнениям.

Показывать:

- таблицу истории;
- график веса;
- график повторений;
- лучший результат;
- динамику относительно предыдущего выполнения.

Цвета:

- рост — зелёный;
- падение — красный;
- без изменений — нейтральный.

Не делай основной метрикой суммарный вес или общий объём тренировки.

## 10. Импорт, экспорт, backup

Обязательно:

- JSON schemaVersion;
- type;
- exportedAt;
- payload;
- валидация перед импортом;
- понятная ошибка при невалидном файле;
- backup восстанавливает весь локальный профиль.

CSV использовать только как дополнительный экспорт для просмотра таблиц.

## 11. Тестирование

Для каждой значимой доменной функции добавляй тесты.

Минимум:

- analytics calculation tests;
- import validation tests;
- repository tests;
- key component tests;
- smoke test запуска приложения.

Перед завершением задачи выполняй:

```bash
npm run lint
npm run typecheck
npm run test
```

Если команда не может быть выполнена, укажи причину и что нужно исправить.

## 12. Стиль кода

- Маленькие файлы.
- Явные имена.
- Никакой магии.
- Одна ответственность на модуль.
- Не смешивать UI, storage и domain logic.
- Не дублировать типы.
- Не создавать универсальные абстракции раньше времени.
- Не использовать `<canvas>`.
- Не добавлять web-only решения, несовместимые с React Native Android.

## 13. Работа с изменениями

Если задача маленькая:

- показывай только изменённые файлы или diff.

Если задача большая:

- создавай/редактируй файлы полностью.

Разделяй несвязанные изменения.

Не предлагай несколько альтернатив одновременно, если можно выбрать одно качественное решение.

## 14. Definition of Done

Задача считается завершённой, если:

- код компилируется;
- типы проходят проверку;
- тесты проходят;
- структура не нарушает FSD;
- функциональность соответствует PRODUCT_SPEC;
- не добавлены лишние функции;
- README или документация обновлены, если изменился запуск или архитектура.

## 15. Приоритеты решений

При конфликте решений выбирай в таком порядке:

1. Простота использования во время тренировки.
2. Сохранность данных.
3. Чистая архитектура.
4. Тестируемость.
5. Возможность масштабирования.
6. Визуальная красота.

Красивый UI не должен усложнять ввод данных.
